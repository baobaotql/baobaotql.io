# 银龄金融安全助手

面向老年群体的适老化 AI 金融反诈风险识别微信小程序（前端）。

## 技术栈

- UniApp + Vue 3
- 微信小程序端

## 功能模块（当前为前端 Mock 版）

| 模块 | 说明 |
|------|------|
| 首页 | 拍照识别、相册上传、历史记录、帮助中心入口 |
| 识别流程 | 选图 → 模拟 OCR → 模拟 AI 分析 → 结果页 |
| 结果页 | 风险等级、五项输出、高危词高亮、语音播报 UI |
| 历史记录 | 本地存储、单条/全部删除、详情回看 |
| 帮助中心 | 使用教程、防骗知识库 |
| 隐私授权 | 首次使用弹窗确认 |

## 适老化设计

- 正文 ≥ 18px（36rpx），标题/按钮 ≥ 24px（48rpx）
- 绿/黄/红三色风险等级，高对比配色
- 大按钮、宽间距、极简单页交互
- 结果页自动触发语音播报（微信同声传译插件 TTS）

## 快速开始

### 1. 安装依赖

```bash
npm install
```

### 2. 配置微信小程序 AppID

编辑 `src/manifest.json`，在 `mp-weixin.appid` 填入你的小程序 AppID。

### 2.1 开通语音播报插件（必做）

语音播报依赖微信官方「**微信同声传译**」插件，需在微信公众平台添加：

1. 登录 [微信公众平台](https://mp.weixin.qq.com/)
2. **设置** → **第三方设置** → **插件管理** → **添加插件**
3. 搜索「**微信同声传译**」并添加（AppID：`wx069ba97219f66d99`）
4. 重新编译小程序后，在微信开发者工具中**重新导入** `dist/dev/mp-weixin`

> 注意：该插件通常要求**非个人主体**小程序；测试号可能无法使用，会提示添加插件。

### 3. 开发运行

```bash
npm run dev:mp-weixin
```

用微信开发者工具打开 `dist/dev/mp-weixin` 目录。

### 4. 生产构建

```bash
npm run build:mp-weixin
```

### 匹配规则（零 token 消耗）

| 匹配强度 | 条件 | 动作 |
|----------|------|------|
| **强匹配** | OCR 命中案例关键词 ≥2 个 | **强制升级「高风险」** + 展示案例 |
| **弱匹配+高风险** | 命中 1 个 + 大模型已判 high | 确认匹配，展示案例 |
| **弱匹配-仅提示** | 命中 1 个 + 大模型未判 high | 展示案例，不升级风险 |

### 示例

用户收到一条短信："您的医保卡已被锁定，请点击链接验证，输入银行卡号和验证码"

→ 大模型 OCR + 分析 → 关键词扫描命中"医保""验证码""银行卡""点击链接"
→ 案例库匹配到 **case_017「医保卡停用恐吓短信」**（命中 4 个关键词）
→ 强匹配 → 风险升级为「高风险」→ 结果页红色警示卡展示真实案例

### 如何新增案例

只需在 `src/utils/caseLibrary.js` 的 `CASES` 数组中追加：

```js
{
  id: 'case_021',
  title: '新案例标题',          // ≤15字
  brief: '案例简要描述…',       // ≤80字
  keywords: ['关键词1', '关键词2', '关键词3'],
  category: '分类名',
}
```

## 完整识别流水线（`analyzeImage`）

| 步骤 | 做什么 | 谁执行 |
|------|--------|--------|
| **Step 1** | 图片 → Base64 | 本地 |
| **Step 2** | 查缓存 | 本地 |
| **Step 3** | **OCR + 风险分析** | **智谱 GLM-4.6V 大模型** |
| Step 4 | OCR 文本截断 | 本地 |
| Step 5 | 低风险安全语言过滤 | 本地 |
| **Step 6** | **案例库关键词匹配** | **本地（零 token）** |
| Step 7 | 缓存结果 | 本地 |

---

## 关键结论

**先走大模型判断风险等级，再走案例库匹配。**

1. **Step 3 大模型优先** — GLM-4.6V 根据 System Prompt 中的风险判定标准（即你 `constants.js` 里的 `HIGH_RISK_KEYWORDS` / `CAUTION_KEYWORDS` 精神），直接输出 `riskLevel`（low/caution/high）

2. **Step 6 案例库后置调权** — `matchCase()` 在本地做关键词匹配，然后：
   - **强匹配（命中≥2个案例关键词）→ 强制升级为高风险**，即使大模型判的低风险也会被覆盖
   - **弱匹配+大模型已判 high → 确认展示案例**
   - **弱匹配+大模型未判 high → 仅提示，不升级**

---

## 补充说明

`constants.js` 里的 `HIGH_RISK_KEYWORDS` / `CAUTION_KEYWORDS` 数组**不会在主流程被直接调用**（`scanRiskKeywords()` 只在轻量模式和大模型失败兜底时使用）。它们是作为**大模型 System Prompt 的知识来源**注入给 GLM 的，告诉它该识别什么。而案例库用的是**自己独立的案例关键词**（每个案例自带 `keywords` 数组），两套关键词体系互不干扰。

简单说：**大模型做初判 → 案例库后置调权，案例库的强匹配可以覆盖大模型的判断。**
